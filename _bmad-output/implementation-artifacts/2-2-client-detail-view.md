# Story 2.2: Client Detail View

Status: done

## Story

As a commercial team member,
I want to view the complete details of a client by selecting them from the list,
so that I can review all their information without navigating away from the clients section.

## Acceptance Criteria

1. **Given** the client list is displayed, **When** the user clicks on a client item, **Then** the right panel shows the complete client details: Nombre, NIT/RUC, Teléfono, Ciudad, **And** the URL updates to `/clientes/:clienteId` (FR30 deep linking).

2. **Given** the user is on the client detail view, **When** the user accesses the URL `/clientes/:clienteId` directly, **Then** the correct client details are loaded and displayed from the backend (`GET /api/v1/clientes/:id`) without requiring the list to be pre-loaded in cache (FR30).

3. **Given** a `clienteId` in the URL does not exist (backend returns 404), **When** the page loads, **Then** a graceful not-found message is displayed in the right panel without crashing the shell layout or triggering a JS error boundary.

4. **Given** the backend is unavailable when loading client detail, **When** the `GET /api/v1/clientes/:id` request fails, **Then** an `ErrorPanel` with a "Reintentar" button is displayed in the right panel (reuses shared `ErrorPanel` component).

5. **Given** the detail panel is rendered, **When** the data is loading, **Then** skeleton placeholders (`react-loading-skeleton`) are shown — NOT a spinner.

## Dev Notes

### Architecture Patterns

This story implements the **detail view layer** of the split-panel client management screen, extending the foundation laid in Story 2.1. The frontend module path follows Clean Architecture: `src/modules/crm/clientes/{domain|application|infrastructure|presentation}/`.

**Route structure (TanStack Router file-based):**
```
frontend/src/routes/_app/clientes.tsx          ← parent layout (split panel, from Story 2.1)
frontend/src/routes/_app/clientes.$clienteId.tsx ← child route rendered inside <Outlet />
```

The `$` prefix in TanStack Router denotes a dynamic segment. `clienteId` is the param name — access via `Route.useParams()`.

**Component tree:**
```
ClientesPage (clientes.tsx)
  ├── ClienteListPanel (280px left panel) ← modified to wire click → navigate
  └── <Outlet />
        └── ClienteDetailPage (clientes.$clienteId.tsx)
              └── ClienteDetailPanel (reads clienteId from route params)
                    └── useCliente(id) → GET /api/v1/clientes/:id
```

**Deep linking strategy (FR30):**
When the user lands directly on `/clientes/:clienteId`, `ClienteDetailPanel` fetches the single client via `useCliente(id)` which calls `GET /api/v1/clientes/:id`. This is independent of the `['clientes']` list cache — no dependency on the list being pre-loaded.

**TanStack Query key for single client:**
```typescript
// useCliente.ts
export function useCliente(id: string) {
  return useQuery({
    queryKey: ['clientes', id],
    queryFn: () => clienteApiRepository.getById(id),
    enabled: !!id,
  })
}
```

**Navigation from list item:**
Navigation is implemented via an `onSelectCliente` callback prop to `ClienteListPanel` (to preserve testability without a Router context). The route passes `useNavigate` and the URL param down as props.

**Not-found handling in `ClienteDetailPanel`:**
Axios throws on 4xx/5xx by default. A 404 is detected via `axios.isAxiosError(error) && error.response?.status === 404` to render a distinct graceful "Cliente no encontrado." message vs. the generic `ErrorPanel` for network failures.

### Backend: `GET /api/v1/clientes/{id}` Endpoint Pattern

```csharp
group.MapGet("/{id:guid}", async (
    Guid id,
    [FromServices] GetClienteByIdQueryHandler handler,
    CancellationToken ct) =>
{
    var dto = await handler.HandleAsync(new GetClienteByIdQuery(id), ct);
    return dto is not null ? Results.Ok(dto) : Results.NotFound();
});
```

### Previous Story Learnings (from Story 2.1)

- **Package manager**: `pnpm` exclusively — never `npm` or `yarn`
- **siesa-ui-kit**: private registry may be unavailable; build custom components as fallback
- **TypeScript strict mode**: `"strict": true` — no `any` types; use `Cliente` not `any`
- **TailwindCSS v4**: use utility classes directly in JSX; `@import "tailwindcss"` in `src/style.css`
- **`routeTree.gen.ts`**: auto-generated by `@tanstack/router-plugin` — the plugin regenerated it automatically when `clientes.$clienteId.tsx` was created
- **dotnet SDK**: not available in sandbox; backend tests noted as complete but not runnable
- **All user-facing text in Spanish**: enforced by code-review
- **Split-panel `<Outlet />`**: the right panel in `clientes.tsx` already renders `<Outlet />` — the new child route renders inside it
- **Router context in tests**: `useNavigate`/`useParams` require Router context; use callback props pattern to keep components testable without router
