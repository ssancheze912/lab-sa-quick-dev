# Story 1.2: Frontend Navigation Shell

Status: ready

## Story

As a user,
I want a persistent navigation structure to access the Clientes and Contactos sections of the application,
so that I can move between sections without full page reloads from any device.

## Acceptance Criteria

1. **Given** the application is loaded on a desktop browser (viewport ‚Č• 1024px), **When** the user views the app, **Then** a `NavigationRail` (siesa-ui-kit) is visible on the left side with "Clientes" and "Contactos" entries, and clicking either entry navigates to `/clientes` or `/contactos` without a full page reload (FR28).

2. **Given** the application is loaded on a mobile browser viewport (< 1024px), **When** the user views the app, **Then** a mobile-responsive `NavigationBar` (siesa-ui-kit) is displayed at the bottom instead of the rail, and all navigation items are visible and tappable with minimum 44px touch targets (FR29).

3. **Given** the user types `/clientes` or `/contactos` directly in the browser URL bar, **When** the page loads, **Then** the correct view is rendered and the corresponding navigation item is active ‚ÄĒ no redirect to a home screen occurs (FR30).

4. **Given** the user navigates to any unknown route (e.g. `/unknown`), **When** the page loads, **Then** a 404 not-found view is displayed with a Spanish message and a link back to `/clientes`.

5. **Given** the navigation shell is rendered, **When** a screen reader or keyboard user navigates the app, **Then** all navigation items have Spanish `aria-label` attributes, are reachable via Tab, and activate via Enter/Space (WCAG 2.1 AA).

## Dev Notes

### Routing Architecture

- **Router:** TanStack Router file-based routing. The Vite plugin (`@tanstack/router-plugin/vite` configured in `vite.config.ts` in Story 1.1) auto-generates `src/routeTree.gen.ts` whenever route files change.
- **Root route file:** `src/routes/__root.tsx` ‚ÄĒ already created in Story 1.1 as a shell placeholder. This story replaces the placeholder content with the full LayoutBase shell.
- **Route files:** Each route file exports a `Route` created with `createFileRoute('/clientes')` and a default component.

**Route file pattern:**
```typescript
// src/routes/clientes.tsx
import { createFileRoute } from '@tanstack/react-router'

export const Route = createFileRoute('/clientes')({
  component: ClientesPage,
})

function ClientesPage() {
  return (
    <div className="p-6">
      <h1 className="text-4xl font-bold tracking-tight">Clientes</h1>
    </div>
  )
}
```

### Navigation Shell Pattern

The UX spec (Direction F) mandates the siesa-ui-kit `LayoutBase` shell with:
- `NavigationRail` (72px collapsed, icon-only) on desktop ‚Č• 1024px
- `NavigationBar` (bottom bar) on mobile < 1024px

**`__root.tsx` pattern:**
```typescript
// src/routes/__root.tsx
import { createRootRoute, Link, Outlet, useRouterState } from '@tanstack/react-router'
import { LayoutBase, NavigationRail, NavigationBar } from 'siesa-ui-kit'
import { UsersIcon, UserIcon } from '@heroicons/react/24/outline'

const navItems = [
  { label: 'Clientes', icon: UsersIcon, to: '/clientes' as const },
  { label: 'Contactos', icon: UserIcon, to: '/contactos' as const },
]

export const Route = createRootRoute({
  component: RootLayout,
  notFoundComponent: NotFoundPage,
})

function RootLayout() {
  const router = useRouterState()
  const currentPath = router.location.pathname

  return (
    <LayoutBase
      navigationRail={
        <NavigationRail
          items={navItems.map(item => ({
            ...item,
            active: currentPath.startsWith(item.to),
            'aria-label': `Ir a ${item.label}`,
          }))}
        />
      }
      navigationBar={
        <NavigationBar
          items={navItems.map(item => ({
            ...item,
            active: currentPath.startsWith(item.to),
            'aria-label': `Ir a ${item.label}`,
          }))}
        />
      }
    >
      <Outlet />
    </LayoutBase>
  )
}

function NotFoundPage() {
  return (
    <div className="flex flex-col items-center justify-center min-h-[60vh] gap-4 p-6">
      <h1 className="text-4xl font-bold tracking-tight">P√°gina no encontrada</h1>
      <p className="text-base text-slate-500">La ruta que buscas no existe.</p>
      <Link to="/clientes" className="text-primary-600 hover:underline font-medium">
        ‚Üź Ir a Clientes
      </Link>
    </div>
  )
}
```

> **Note:** If `siesa-ui-kit` `LayoutBase` / `NavigationRail` / `NavigationBar` component API differs from the pattern above, adapt the props to match the actual exported API. Always check `node_modules/siesa-ui-kit` exports before assuming prop names.

### Active Navigation State

TanStack Router `<Link>` supports `activeProps` for styling the active route:
```typescript
<Link
  to="/clientes"
  activeProps={{ className: 'bg-primary-50 text-primary-700' }}
  inactiveProps={{ className: 'text-slate-600 hover:bg-slate-100' }}
>
  Clientes
</Link>
```
If siesa-ui-kit nav components accept an `active` boolean prop, pass `active={currentPath.startsWith(item.to)}`.

### Responsive Breakpoint

Per company standards and UX spec:
- Desktop: `lg:` breakpoint = 1024px (Tailwind default)
- `hidden lg:block` on `NavigationRail` container
- `block lg:hidden` on `NavigationBar` container

### 404 Handling

TanStack Router supports a `notFoundComponent` at the root route level. It catches any unmatched route and renders the specified component. Register it in `createRootRoute({ notFoundComponent: NotFoundPage })`.

Alternatively, create a `src/routes/$.tsx` catch-all file route:
```typescript
// src/routes/$.tsx
import { createFileRoute } from '@tanstack/react-router'
export const Route = createFileRoute('/$')({ component: NotFoundPage })
```

Prefer the `notFoundComponent` approach on the root route as it avoids polluting the route tree with a dynamic segment.

### Placeholder Views

`ClientesPage` and `ContactosPage` in this story are stubs only ‚ÄĒ they render the section title and nothing else. The full client list and contact list UI are implemented in Epics 2 and 3 respectively.

### Testing Setup

Tests use Vitest + React Testing Library. The test environment is `jsdom` (configured in `vite.config.ts` from Story 1.1). Import `@testing-library/jest-dom` matchers via `src/test-setup.ts`.

For router-aware tests, wrap the component under test in a `RouterProvider` with a `createMemoryHistory`:
```typescript
import { createMemoryHistory, createRouter } from '@tanstack/react-router'
import { routeTree } from '../../routeTree.gen'

const memoryHistory = createMemoryHistory({ initialEntries: ['/clientes'] })
const router = createRouter({ routeTree, history: memoryHistory })
render(<RouterProvider router={router} />)
```
